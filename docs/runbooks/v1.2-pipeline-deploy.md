# Deploying v1.2 — resilient COM/SOM pipeline

> **Audience:** operators upgrading a running BSS-CLI deployment from any v1.1.x
> to v1.2. This release converts COM, SOM, and subscription to a transactional
> outbox + safe (DLQ + inbox) consumers. The code change is invisible at the API
> surface but it introduces a **one-time queue-redeclaration migration** that
> needs an explicit deploy step — skipping it lands the new containers without
> any consumer attached.

## Why this is not a normal redeploy

v1.2's consumers declare each queue with `x-dead-letter-exchange` arguments so a
handler failure routes to a retry queue instead of being dropped. RabbitMQ
refuses to redeclare an existing durable queue with different `arguments` —
`PRECONDITION_FAILED - inequivalent arg 'x-dead-letter-exchange'`. The new
container's `setup_consumer` catches that error and logs `mq.consumer.setup_failed`,
but **the consumer never attaches**. Orders would stage `service_order.completed`
events the relay delivers correctly, then sit in the unconsumed queue forever.

There is no in-RabbitMQ migration for queue arguments — the queue must be
deleted and re-declared. Since the order pipeline drains its queues in seconds
under normal load, the six affected queues are empty at any quiet moment, which
is when the migration runs.

## Migration procedure

```bash
# 0. Pre-flight: confirm no in-flight orders. Each affected queue must be empty.
#    BSS_MQ_URL host is the RabbitMQ management endpoint on :15672.
MQ_HOST=tech-vm           # or rabbitmq, or your broker host
MQ_USER=admin             # the user from BSS_MQ_URL
MQ_PASS=...
for q in \
    com.service_order.completed \
    com.service_order.failed \
    som.order.in_progress \
    som.provisioning.task.completed \
    som.provisioning.task.failed \
    som.provisioning.task.stuck; do
  n=$(curl -s -u "$MQ_USER:$MQ_PASS" \
      "http://$MQ_HOST:15672/api/queues/%2F/$q" \
      | grep -oE '"messages":[0-9]+')
  echo "$q -> $n"
done
# Every line must show "messages":0 before proceeding. If any is non-zero, wait
# for it to drain (the running v1.1 consumer will work it off), or stop new
# order intake first.

# 1. Apply the schema migration (additive + reversible).
make migrate         # alembic upgrade 0026 -> 0027

# 2. Build the new images (com, som, subscription).
docker compose build com som subscription

# 3. Delete the six legacy queues with the if-empty guard. 204 = deleted.
#    The guard means an unexpectedly non-empty queue refuses to delete instead
#    of losing messages.
for q in \
    com.service_order.completed \
    com.service_order.failed \
    som.order.in_progress \
    som.provisioning.task.completed \
    som.provisioning.task.failed \
    som.provisioning.task.stuck; do
  code=$(curl -s -o /dev/null -w "%{http_code}" \
      -u "$MQ_USER:$MQ_PASS" -X DELETE \
      "http://$MQ_HOST:15672/api/queues/%2F/$q?if-empty=true&if-unused=true")
  echo "DELETE $q -> $code"
done

# 4. Recreate the containers. They'll redeclare each queue WITH the
#    x-dead-letter-exchange args, plus the new <queue>.retry / <queue>.parked
#    siblings.
docker compose up -d --no-deps com som subscription

# 5. Confirm the new topology + that consumers attached.
for s in com som subscription; do
  echo "── $s ──"
  docker compose logs $s 2>&1 | grep -E "outbox.relay.started|mq.consumer.started" | tail
done
# You want to see (a) outbox.relay.started for all three, and (b) one
# mq.consumer.started per declared queue (com=2, som=4, subscription=2).
```

## Post-deploy smoke

```bash
# 1. Drive one real order end-to-end through the rebuilt pipeline.
bss order create --customer <CUST-with-COF> --offering PLAN_S --wait
# Expected: state goes acknowledged → in_progress → completed within seconds.

# 2. Confirm the relay delivered (not inline). All COM/SOM events for that
#    order should show published_to_mq=true with published_attempts >= 1.
psql "$BSS_DB_URL_PSQL" -c "
  SELECT event_type, published_to_mq, published_attempts
  FROM audit.domain_event
  WHERE payload->>'commercialOrderId' = '<ORDER_ID>'
    AND event_type IN ('order.in_progress','service_order.completed','order.completed')
  ORDER BY occurred_at;"

# 3. Confirm the subscription's commercial_order_id idempotency key is set.
psql "$BSS_DB_URL_PSQL" -c "
  SELECT id, commercial_order_id, state FROM subscription.subscription
  WHERE commercial_order_id = '<ORDER_ID>';"
```

## Rollback

The migration `0027` is additive and reversible. To roll back:

```bash
# 1. Stop com, som, subscription.
docker compose stop com som subscription

# 2. (Optional) delete the new DLX/parked/retry queues. They become dead weight
#    under v1.1 code but are harmless to leave.
#      <queue>.retry, <queue>.parked, exchange "bss.events.retry"

# 3. Restore the v1.1 images + recreate.
docker compose up -d --no-deps com som subscription

# 4. Downgrade the schema.
cd packages/bss-models && uv run --package bss-models alembic downgrade 0026

# Under v1.1 code the main queues will still have the DLX argument (declared by
# v1.2). v1.1's setup_consumer redeclares without arguments → PRECONDITION_FAILED
# → consumer doesn't attach. Symmetric problem to step 3 of the forward
# migration. Solution is identical: delete the six queues empty, restart v1.1
# containers, they'll redeclare without arguments.
```

## What to watch for after deploy

| Signal | Meaning | Action |
|---|---|---|
| `mq.message.parked` in com/som logs | A handler exhausted its retry budget. The message is in `<queue>.parked` — not lost. | Inspect the parked message body (`curl -X POST .../get`), find the root cause, fix, and republish from the parked queue. |
| `order.stuck` events | The reconciliation sweeper found an order `in_progress` past 15 min. | Triage the order with `bss order show <id>` — usually a parked message is the upstream cause. |
| `outbox.relay.publish_failed` | A row published_attempts is climbing and `last_publish_error` is set. | RabbitMQ is unreachable from that service. Relay will keep retrying — fix MQ connectivity. |
| `inbox.duplicate.skipped` | A redelivery was correctly deduped. | Normal. Indicates the inbox is doing its job. |
