# 04 — Risks, Open Decisions, Estimate

## 1. Risk register (ranked)

**R1 — Silent behavior drift at dict-shaped seams.** Pydantic coerced, `.get()` defaulted,
and dict payloads absorbed shape variance that serde will reject or encode differently
(datetime formats, numeric coercion, absent-vs-null, camelCase families). *This is the #1
source of bugs in any Python→Rust port.* Mitigation: the golden-contract rig (Phase 0) diffs
real request/response/event JSON per endpoint before any cutover; TMF payloads get
spec-shaped serde tests; the csr `field()` camel/snake dual-reader gets an explicit typed
translation table.

**R2 — LLM behavior parity is not diffable.** Same prompts + same tools ≠ same transcripts;
tool JSON schemas generated by schemars will differ cosmetically from langchain's inference,
and models are sensitive to schema wording. Mitigation: keep tool descriptions/param
descriptions byte-identical (they come from docstrings/Annotated hints — port as string
constants); fixture-driven e2e replays for the deterministic layer; human-reviewed live soak
corpus runs for the judgment layer; do not tune prompts and port simultaneously.

**R3 — Double-billing / quota-math regressions in subscription.** The renewal sweeps'
mark-before-dispatch, SKIP LOCKED semantics, and FOR UPDATE balance decrements encode
hard-won correctness (the repo documents the identity-map double-write bug). Mitigation:
port raw SQL verbatim; proptest port of the hypothesis suite *before* the service logic;
a dedicated concurrent-renewal integration test (two workers, one Postgres) that Python
never had.

**R4 — Auth material incompatibility bricks sessions/tokens at cutover.** Three HMAC schemes
with different keying; portal sessions and login tokens are *rows* that must verify across
the language boundary during Phase 6 canarying. Mitigation: golden vectors captured from
Python in Phase 0 (token hash, OTP hash, all three webhook schemes, step-up tokens) are
non-negotiable conformance tests; cutover flushes nothing.

**R5 — Scope creep via "while we're here."** A rewrite invites fixing the vestigial
`transitions` dep, the missing CI, the non-distroless images, mypy-strict debt… Some of that
is *in* this plan deliberately (CI in P0, distroless in P8). Everything else: the port
changes **zero behavior** until Phase 8. New features during the migration go to the Python
repo first (it's the oracle) or wait. This needs to be an explicit working agreement.

**R6 — Ecosystem gaps discovered late.** Known-thin spots: async-stripe maintenance (avoided
via D4), OTel-tracing version-matrix churn, MiniJinja filter parity, reedline maturity.
Mitigation: every externally-risky crate is exercised in Phase 0/1 (that's partly what the
pilot is for); each has a hand-roll fallback noted in doc 02.

**R7 — The oracle rots.** If the Python repo drifts (new features land there) the golden
tests chase a moving target. Mitigation: feature-freeze windows per phase, or (better) treat
Python `main` as frozen at a tagged parity baseline and rebase golden captures explicitly and
deliberately.

**R8 — Solo-maintainer bus factor & morale.** 55–77 pw of porting is a grind with no
user-visible payoff until late. The Phase 4 bilingual resting point is the pressure valve:
a coherent, shippable, half-migrated system where the project can pause indefinitely.

### Resolved seam — the eSIM ASCII QR is not byte-identical (2026-07-15, P6c s2)

**Decided: accept the divergence.** `render_esim_activation` draws an ASCII QR of the
LPA string. python-qrcode and Rust's `qrcode` crate encode the *same* payload into
**different matrices**: they segment the string into QR modes differently (so the
bitstream → codewords → modules differ) **and** they choose different mask patterns.
Forcing the mask is not sufficient — verified by driving the `qrcode` crate's canvas
API directly with mask 7 (python's pick) and still getting a different matrix, which
proves the divergence is in the data encoding, not just masking.

Both outputs are **valid QR codes that scan to the identical LPA string**, and both
libraries select the same *version* for a given payload, so the block's dimensions —
and hence the card's line count and frame — are unchanged. The divergence is confined
to which cells inside the square are dark.

The alternative was reimplementing python-qrcode's segmentation optimizer + its
`lost_point` mask-penalty scoring on top of the `qrcode` crate's canvas — a few
hundred lines of brittle mimicry of another library's internals, needing its own
golden corpus to trust, to make a scannable square render identical pixels.

**How the test reflects it:** `esim_card_matches_the_oracle_outside_the_qr_block`
asserts **byte equality on every non-QR line** (frame, labels, the last-4 redaction,
the LPA text) and asserts the QR block's **functional** contract only (same line
count, same width, only block glyphs). No assertion claims parity the port doesn't
have. This is the one place in the port where "byte-identical to the oracle" is
knowingly not the standard — everywhere else it still is.

## 2. Open decisions (need a human call)

**D1 — Go/no-go framing: what is the port *for*?** The plan assumes whole-system properties
(static binaries, footprint, type-safe policies, longevity) justify 12–18 months. If the
actual driver is narrower (e.g. just the service plane's footprint), stop at the Phase 4
resting point by design and keep portals/orchestrator/CLI in Python permanently — that's a
legitimate end state this plan supports. **Recommended: decide the intended end state before
Phase 0, revisit at Phase 4.**

**D2 — Migration authority during transition.** Recommended (and assumed): Alembic remains
the only schema authority until Phase 8; Rust never migrates; sqlx compile-time checks run
against an Alembic-migrated database. Alternative (rejected): dual migration tools — a
coordination hazard for zero benefit.

**D3 — Orchestrator port order.** Recommended (and assumed): orchestrator as a Rust lib
before portals (P5→P6), preserving the in-process doctrine. Alternative: temporary HTTP
sidecar exposing `astream_once` so portals could port first — rejected: builds a throwaway
network seam the doctrine forbids, and SSE-over-SSE relaying is fiddly.

**D4 — Stripe client.** Recommended: direct reqwest against the 5–6 Stripe endpoints used,
keeping the SAQ-A surface small and auditable. Alternative: async-stripe crate — revisit if
the payment phase finds the surface bigger than inventoried.

**D5 — Tool arg schema fidelity. (resolved 2026-07-13.)** Option A (recommended): typed
structs + schemars, accept cosmetic schema drift, validate against the fixture corpus.
Option B: replay langchain's exact JSON schemas as embedded constants for bit-parity,
migrate to schemars later. **Resolution: Option A — schemars.** Derive each tool's JSON
Schema from a typed Rust arg struct; the tool *description* + *param docstrings* stay
byte-identical string constants (they drive model behaviour — R2), but the schema envelope
is generated. Behaviour is validated against the fixture corpus + human-reviewed live soak,
not by bit-diffing the schema. Revisit only if a fixture/soak run shows the model's
tool-call behaviour shifting on the schema shape.

**D6 — Team shape.** Solo (12–18 mo) vs pair from Phase 2 (7–10 mo wall-clock). The plan is
written to make the pair split clean (playbook after P1, per-service independence).

**D7 — Repo home.** This directory (`bss-cli-rust`) becomes the Cargo workspace root
(recommended — clean history, plan docs move to `docs/migration/`), vs a `rust/` subtree in
the existing monorepo (easier shared assets/scenarios during transition, messier archive
story). If staying separate, scenario YAML + templates + prompts are *copied* at each phase
with a checksum check against the oracle, not symlinked.

**D8 — When does a service actually cut over? (resolved 2026-07-11.)** Two plan statements read
as a conflict: "retire the Python container from compose" *at each phase* (§03 opening, §1
per-service cutover) vs "the Python oracle must stay runnable until P8" (§03 parallelization).
**Resolution: per-service cutover into the *running* stack, at each service's own phase**
(P1–P4), because it surfaces integration errors at one-service blast radius while the cost to
fix is lowest — not deferred to a Phase-8 big bang. "Oracle stays runnable" means *reproducible
on demand*, not *primary running container*: the Python image build stays in the base
`docker-compose.yml`, and the Rust container is layered on via the `docker-compose.rust.yml`
overlay (`docker compose -f docker-compose.yml -f docker-compose.rust.yml up -d <svc>`).
Dropping the overlay brings the Python oracle back for a golden diff. **Phase 8 is the *final*
cutover** — Alembic freeze, image hardening, archiving the Python repo, the 14-day soak — i.e.
when the *last* Python service is retired and the tree is decommissioned, not the first time
Rust runs. The overlay's "cut over so far" list is the running ledger.

**D9 — JSON object key ordering: `serde_json` `preserve_order`. (resolved 2026-07-13, P5c slice 4.)**
Python preserves insertion order for dicts everywhere (Pydantic/FastAPI response bodies,
`json.dumps`, `httpx.json()` round-trips). Rust's default `serde_json::Value` is backed by a
`BTreeMap`, so re-serialization **sorts keys alphabetically**. This was invisible through P1–P4
because every service golden diff compares parsed `Value`s (order-independent — see
`services/*/tests/live_smoke.rs::get_json`). It first *bites* in the orchestrator (P5c): the
ReAct loop feeds each tool result to the model as `Value::to_string()`, so a sorted-key
observation would diverge from the Python transcript the R2 fixture-corpus parity gate replays,
and **projected-dict** tools (`subscription.get_esim_activation` builds a fixed-key dict) would
emit visibly-reordered JSON. **Resolution: enable `preserve_order` workspace-wide** (root
`Cargo.toml`, `serde_json = { features = ["preserve_order"] }`; the feature unifies across the
graph). `Value`'s map becomes an `IndexMap`, so parse→reserialize and `json!` both preserve
insertion order — matching Python everywhere at once (services *and* tool observations). Verified
**zero test breakage**: the whole workspace stays green (the golden diffs are `Value ==`, so
unaffected; nothing string-compares serialized JSON with a sorted-key expectation). This
supersedes the ad-hoc P5b workaround rationale (pending-destructive arg order was preserved via a
manual `IndexMap` + `py_repr` read of the `json` text column — still correct, just no longer the
*only* order-preserving path). Behaviour-frozen holds: JSON key order is not external behaviour
(consumers parse), and the running Rust containers only change wire ordering on their next build.

## 3. Estimate & error bars

**55–77 person-weeks** across phases (doc 03). Basis: ~87k LOC of non-test Python that
actually ports (109k minus tests-kept-as-reference, e2e, dead code), against a typical
15–20 productive-port-LOC-equivalent/hour for contract-frozen rewrites with a good harness —
then sanity-checked bottom-up per phase. Rust LOC outcome: expect ~1.1–1.4× the Python
(types + error handling, minus ORM/pydantic magic), i.e. **~120–150k LOC**.

What the error bars hinge on: R1 (seam drift) dominates services; R2 (LLM parity) dominates
P5–P7; template parity dominates P6. The pilot phase exists to convert these unknowns into
measured rates — **re-estimate the whole plan after Phase 1** using actual rating-service
throughput; that retro is the single most valuable estimation event in the project.

What is *not* in the estimate: new features, prompt tuning, multi-tenant work, real NE
adapters, the deferred billing service — the port is behavior-frozen by rule R5.
