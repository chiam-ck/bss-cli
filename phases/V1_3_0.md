# v1.3.0 — Customer↔offer pairing in loyalty at assign time

**Status:** shipped 2026-05-25. Bumps `BSS_RELEASE` to 1.3.0; alembic head 0029.

## What changed

`bss promo assign` (the targeted-promo eligibility-list write) now **also mints
the customer↔offer pairing in loyalty upfront** via `offer.issue`, so loyalty's
per-customer views reflect the assignment the moment the operator runs it.
Activation switches from "mint-and-claim by code" to "advance the pre-paired
offer" for the targeted lane. Public typed codes are unaffected — they still
mint-and-claim by code at activation.

## Why

The v1.1.1 amendment (2026-05-22) collapsed consume to **one path** —
`offer.claim` by code — and retired `offer.issue` / `offer.advance_to_claimed`.
At the time the trade-off favoured path-collapsing simplicity. In operation
the cost became clear: BSS holds eligibility but loyalty doesn't know which
customers are paired with a targeted promo until they place an order.
Operator views in loyalty (per-customer offers, campaign rosters) show empty
slots that BSS thinks are full — auditability gap, reconciliation cost. CK
called this out 2026-05-25: *"I need the pairing to be done upfront."*

The v1.3.0 reversal restores upfront pairing **for the targeted lane only**.
Public codes were never about pairing — they're consumer-typed at signup —
so they keep the v1.1.1 path.

## Doctrine reversal — explicit

| | v1.1.1 amendment | v1.3.0 |
|---|---|---|
| Targeted, assign-time | BSS eligibility row only | BSS eligibility row + `offer.issue` (saga; idem on deterministic offer id) |
| Targeted, activation | `offer.claim(source=promo_code)` | `offer.advance_to_claimed(offer_id)` |
| Public typed, activation | `offer.claim(source=promo_code)` | unchanged — same path |
| Loyalty-side visibility on assign | Invisible until first order | Visible immediately (per-customer offers, campaign rosters) |

The retired `offer.advance_to_claimed` path is back — but **only** for the
targeted lane. Public codes never go through it.

## Mechanics

1. **`catalog.promotion_eligibility.loyalty_offer_id`** — new nullable column
   (migration 0029). Stores the upfront-minted loyalty offer id; NULL means
   either a pre-v1.3.0 row OR a v1.3.0 row where `offer.issue` degraded.

2. **`PromotionService.assign_targeted`** — for each new (promotion, customer)
   pairing: deterministic offer id `OFF-<customer_id>-<promotion_id>`, call
   `loyalty.issue_offer(...)` with `source={"type": "campaign", "campaign_id":
   <promotion_id>}`, idempotency key = the offer id (so a second `bss promo
   assign` for the same pair is a no-op on loyalty too). Stamp the id on the
   eligibility row.

3. **Saga semantics** — if `offer.issue` refuses / loyalty is unreachable, the
   eligibility row is **still written** with `loyalty_offer_id=NULL` and
   `catalog.promotion.loyalty_issue_failed_degrade` is logged. BSS-side
   eligibility is honoured; we lose only the upfront visibility win. At
   activation, the consume path detects `promo_offer_id IS NULL` and falls
   back to claim-by-code transparently (same behaviour as a pre-v1.3.0 row).
   Pattern: never block the assignment on loyalty (mirrors the create-time
   "loyalty error → full price, never block the order" rule).

4. **`resolve_eligible_promo` + `validate_for_order`** — both now return
   `loyalty_offer_id` (route field `loyaltyOfferId`). NULL means
   "consume falls back to claim-by-code."

5. **`COM order_service._resolve_discount`** — populates `promo_offer_id` on
   the order item from `loyaltyOfferId` at order create. Stamped intent, not
   yet claimed.

6. **`COM order_service._claim_entitlement`** — branches:
   - `item.promo_offer_id` set → `advance_offer_to_claimed(offer_id)` (targeted).
   - `item.promo_offer_id` NULL → `claim_offer(source=promo_code)` (public, or v1.3.0 targeted-degraded backstop).

   Returns the offer id either way; redeem/revoke unchanged.

## Idempotency keys (unchanged; just additions)

| Step | Key |
|---|---|
| Assign / `offer.issue` | `OFF-<customer_id>-<promotion_id>` (= the deterministic offer id) |
| Activate / `advance_to_claimed` (targeted) | `<order_id>:claim` |
| Activate / `claim_offer` (public) | `<order_id>:claim` (unchanged) |
| Redeem | `<order_id>:redeem` |
| Revoke (on activation failure) | `<order_id>:revoke` |

## Backwards compatibility

- **Pre-v1.3.0 eligibility rows** have `loyalty_offer_id = NULL`. Activation
  falls back to claim-by-code transparently — no migration script needed for
  history.
- **Code paths that survive unchanged**: public typed codes; renewal pricing;
  discount snapshot composition; subscription FSM; bss promo create.
- **What this DOES touch**: only `bss promo assign` + the COM
  service_order.completed consumer. Two files, two test classes.

## Also in this release

- **Bug fix bundled (`fix(subscription): partial unique on msisdn/iccid (exclude terminated rows)`)**:
  before v1.3.0 the `UNIQUE (msisdn)` / `UNIQUE (iccid)` constraints on
  `subscription.subscription` blocked inventory from recycling phone numbers
  whose old subscriptions were already terminated — every signup that picked a
  recycled number bricked at `subscription.create` with an IntegrityError after
  KYC + payment had already succeeded, stranding the order in_progress. Replaced
  with partial unique indices that exclude terminated rows (migration 0028).

## Backfill done at release time

The first v1.3.0 deployment carried one operator-driven backfill: chiamck+002
(`CUST-c9ab5608`) had been assigned `PROMO_VIP` before v1.3.0 shipped, so the
upfront pairing was minted by hand via `loyalty.issue_offer` and the
`loyalty_offer_id` column backfilled on that one row (`OFF-CUST-c9ab5608-PROMO_VIP`).
No other rows needed touching (the rest were created at promo-design time, not
operator-assigned).
