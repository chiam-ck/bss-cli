> **The release where the COM/SOM event pipeline stops being able to silently lose or strand an order.** No new customer-facing feature. v1.2 finishes the outbox that migration `0001` already scaffolded but never wired, gives every MQ consumer a retry/dead-letter path instead of "drop on exception", makes the consume handlers idempotent under redelivery, and adds a reconciliation sweeper that surfaces stuck orders to the operator instead of leaving them stranded forever. The v1.1.3 promo bug was one symptom of a structural gap; this release closes the gap.

## Why now

v1.1 added a multi-step distributed saga (claim → subscription.create[charges COF] → redeem) **inside a single MQ handler** that had no retry, no dead-letter, and no idempotency. The v1.1.3 fix wrapped the one line that bricked a paid order. Every other line in `handle_service_order_completed` can still strand an order the same way, because the failure was never in the loyalty call — it was in the pipeline that has no safe failure mode.

## What this is NOT

- **Not a new feature.** Zero new customer surface. The only new operator surface is read/repair tooling for stuck orders.
- **Not a messaging-stack change.** Still RabbitMQ topic exchange `bss.events`, still aio-pika, still `audit.domain_event` as the durable log. We finish the *simplified outbox* doctrine already in CLAUDE.md — we don't adopt Kafka, Debezium, or a CDC pipeline.
- **Not exactly-once.** At-least-once delivery + idempotent consumers (the inbox pattern). Doctrine stays "best-effort delivery backed by the durable audit log" — we make "best-effort" actually mean *eventually delivered*, not *dropped on first failure*.
- **Not a saga framework.** No `transitions`-based saga engine, no Temporal, no Celery. The order FSM already exists; we make its event edges durable.
- **Not motto-violating.** No retries inside payment (Motto / "No retry/dunning"). Charge still fails-closed. What we retry is *event delivery and consumer processing*, never a declined charge.

## Decisions to lock (Phase 0)

1. **Finish the existing outbox; don't replace it.** Keep `audit.domain_event` as the outbox table (the `published_to_mq` flag + partial index are already there). Add a relay, fix the ordering. No new outbox table.
2. **Relay lives as a shared in-process worker** (`packages/bss-events` → `outbox_relay.py`), started in each service's lifespan — same pattern as the v0.18 subscription renewal worker (in-process tick loop, `FOR UPDATE SKIP LOCKED`, no sibling scheduler). One relay per service process, sweeping that service's own schema's unpublished rows. Rationale: keeps the schema-per-service boundary; no cross-schema reader; mirrors a pattern the codebase already trusts.
3. **Publish strictly after commit.** `publish()` only *stages* the audit row (in the txn). The relay is the **only** thing that calls `exchange.publish()`. This removes D2 entirely: no publish path runs before the commit, so a phantom event is impossible. Trade-off: a few-hundred-ms added latency from write→deliver (relay tick interval). Acceptable — these are async reactions, not the synchronous HTTP plane.
4. **Every queue gets a dead-letter exchange + bounded retry.** `bss.events.dlx` fanout → per-queue `*.dlq`. Handler exception → reject to DLX with a retry counter header; after N retries (default 5, env `BSS_MQ_MAX_RETRIES`) park in the DLQ. No infinite redelivery loop.
5. **Inbox dedup at the consumer seam.** A `messaging.processed_event` table per consuming schema, keyed on `event_id`. The consumer checks-and-inserts before side effects; a duplicate is acked and skipped. `event_id` already exists on every published event (`DomainEvent.event_id`) — we just need to put it on the wire and check it.
6. **Reconciliation sweeper is operator-visible, never auto-resolving.** A worker flags orders `in_progress` past `BSS_ORDER_STUCK_THRESHOLD` (default 15 min) and emits `order.stuck` → cockpit surface. It does **not** auto-cancel or auto-complete (that's a policy decision a human/▶`/confirm` makes). Mirrors the provisioning-sim "stuck" doctrine.

## Schema delta

### `audit.domain_event` — no column change, add relay bookkeeping
The flag + partial index already exist. Add:
- `published_attempts SMALLINT NOT NULL DEFAULT 0` — relay increments; lets us alert on a row that won't publish.
- `last_publish_error TEXT NULL` — last failure for `bss trace`/cockpit triage.

Migration `00NN_v120_outbox_relay` in `bss-models`.

### new `messaging.processed_event` (inbox) — per consuming schema
`packages/bss-models`, one table replicated into each consumer's schema (`order_mgmt`, `service_inventory`, `subscription`, …) to keep the boundary:
- `event_id UUID PRIMARY KEY`
- `consumer TEXT NOT NULL` (queue name — same event can fan out to N consumers)
- `processed_at TIMESTAMPTZ NOT NULL`
- PK is `(event_id, consumer)`.

Migration `00NN_v120_inbox`.

## Workstreams

### Track 1 — Transactional outbox relay (closes D1 + D2)
- `packages/bss-events/bss_events/outbox_relay.py`: `start_relay(app)` lifespan helper. Tick loop (env `BSS_OUTBOX_RELAY_INTERVAL_MS`, default 250ms): `SELECT … WHERE NOT published_to_mq ORDER BY occurred_at FOR UPDATE SKIP LOCKED LIMIT 100`, publish each (routing key = `event_type`, body = `payload`, `event_id` in headers), set `published_to_mq=True` + `published_attempts += 1` in the same txn, commit. On publish failure: increment attempts, store `last_publish_error`, leave unpublished → next tick retries.
- `publish()` in **all 7 publishers** (`com`, `som`, `crm`, `subscription`, `payment`, `mediation`, `rating` + `provisioning-sim/worker.py`) drops the inline `exchange.publish()` block — becomes "stage the `DomainEvent` row only". This is a mechanical, identical edit across services → candidate for a shared `bss_events.stage_event()` helper so the publishers converge.
- Lifespan wiring in each service `main.py`/`dependencies.py`: `start_relay(app)` alongside the existing consumer setup. Multi-replica safe via `SKIP LOCKED`.
- ⚠ Ordering note: relay publishes in `occurred_at` order per aggregate; cross-aggregate ordering is not guaranteed (it never was). Consumers must not assume global ordering — document in the doctrine.

### Track 2 — Safe consumers: DLQ + bounded retry (closes D3)
- New `bss_events.consumer` helper wrapping the boilerplate every consumer repeats: declare queue with `arguments={"x-dead-letter-exchange": "bss.events.dlx"}`, parse, inbox-check, run handler, ack. On handler exception: read `x-death` count; `< MAX` → reject to DLX (redelivered after TTL backoff); `>= MAX` → publish to `*.parked` + emit `mq.message.parked` + ack original (stop the loop).
- Rewrite `com/consumer.py` (2 queues) and `som/consumer.py` (4 queues) onto the helper. Same for `subscription`, `crm` consumers.
- `bss.events.dlx` (fanout) + retry queue with `x-message-ttl` (env `BSS_MQ_RETRY_BACKOFF_MS`, default 5000) dead-lettering back to the source queue — the classic delayed-retry topology, no plugin needed.
- A parked message is a **cockpit-visible incident**, not a black hole: `bss mq parked` lists them; `bss mq replay <event_id>` re-injects after a fix.

### Track 3 — Idempotent consume (closes D4)
- Inbox check in the consumer helper (Track 2): `INSERT INTO messaging.processed_event (event_id, consumer) … ON CONFLICT DO NOTHING RETURNING` — no row inserted ⇒ already processed ⇒ ack + skip. Same txn as the handler's writes, committed together, so the side effect and the dedup mark are atomic.
- Make `subscription.create` idempotent on `commercial_order_id` (it's already in the payload) — return the existing subscription if one exists for that order, don't re-charge. This protects the COF even if the inbox is somehow bypassed (defense in depth; the COF charge is the thing we must never double).
- Audit the COM completion saga for re-entrancy now that redelivery is *expected* (it backs the retry path), not just a crash edge: claim (idem key `{order}:claim` ✓ already), `subscription.create` (now idem ✓), redeem (idem ✓). Document the saga's idempotency contract inline.

### Track 4 — Reconciliation sweeper + operator surface (defense in depth)
- `services/com/app/workers/reconciliation.py`: in-process tick (default 60s) — orders `in_progress` older than `BSS_ORDER_STUCK_THRESHOLD` → emit `order.stuck` (once; mark `stuck_flagged_at`). Symmetric SOM sweep for service orders / provisioning tasks with no terminal event.
- Cockpit: `order.stuck` lands on the operator dashboard; `bss order triage` shows the saga's last known step (from `audit.domain_event` for that aggregate) so the operator sees *where* it stalled. Re-uses `bss trace`.
- This is the human backstop for anything the automated retry/DLQ path can't resolve (e.g. a genuinely declined charge mid-saga). It does not auto-act.

## Doctrine additions (CLAUDE.md anti-patterns — Phase 0 amendment)

- **The relay is the only publisher.** No service calls `exchange.publish()` directly; all events go through `stage_event()` + the relay. Grep guard: `exchange.publish(` may appear only in `bss_events/outbox_relay.py`.
- **Every consumer declares a DLX.** A `declare_queue` without `x-dead-letter-exchange` is a doctrine bug. Grep guard.
- **Every consumer is idempotent.** A handler with side effects that doesn't go through the inbox check is a doctrine bug.
- **Stuck ≠ lost.** A stranded order must always be either retrying (DLQ), parked (operator-visible), or flagged (`order.stuck`). "Vanished" is never acceptable.

## Testing

- **Unit:** relay publishes + marks; relay leaves row unpublished on publish failure and retries next tick; publisher stages only (no `exchange.publish` call); inbox skips a duplicate `event_id`; `subscription.create` returns existing sub for a repeated `commercial_order_id` (no second charge).
- **Integration (the gap unit tests miss — recurring lesson):** kill RabbitMQ, fire `order.in_progress`, restart MQ, assert the relay delivers and the order completes. Crash the COM consumer mid-`handle_service_order_completed` (after charge, before commit), redeliver, assert exactly one subscription + one charge.
- **Chaos:** extend the soak harness — inject MQ-down windows and consumer exceptions; soak gate adds "zero orders end stranded" and "zero double-charges". This is the gate that would have caught v1.1.3.
- **Regression:** the exact v1.1.3 exhausted-code path now exercises the DLQ → degrade path, not a dropped message.

## Open questions for CK (decide before build)

1. **Relay placement** — shared in-process worker per service (recommended, matches v0.18) **vs** one standalone `bss-relay` sidecar container reading all schemas. The latter breaks the schema-per-service read boundary; I'd avoid it, but it's one process instead of seven loops.
2. **Retry backoff** — fixed TTL (simple, recommended) vs exponential via multiple TTL queues (more robust, more topology). Start fixed?
3. **Scope of v1.2** — all four tracks, or split: v1.2 = outbox + DLQ (D1–D3, the silent-loss bugs), v1.3 = inbox + reconciliation (D4 + operator surface)? D1–D3 are the "events vanish" class; D4 is the "redelivery double-acts" class. Shipping them together is cleaner but larger.
