# V1.4.0 — End-to-end test suite via Playwright

**Status:** planned (2026-05-25)
**Branch:** `feat/v1.4-e2e-suite` (to be cut)
**Migration head at start:** 0029

## Why now

Through v1.1 → v1.3 we shipped four real fixes — and every one was caught by
hand (screenshot capture, the chiamck+001 stuck-order report, the exhausted-code
brick). The portal funnel has enough surface now (auth, KYC, payment, signup,
promo preview, dashboard, cockpit veneer) that a regression in any of them is
silent until an operator stumbles on it. v1.4 closes that gap.

## Scope (this release)

**Smoke + promo branches**, ~10 specs, ~4 min wall-clock. Two surfaces:

### Self-serve portal (`localhost:9001`)
1. **Signup golden path** — magic-link login → KYC (prebaked) → payment (mock) →
   plan select → MSISDN reserve → activation → dashboard shows active
   subscription + eSIM QR.
2. **Public promo applied at signup** — `DEMO_WELCOME10` typed at the promo
   input → live preview discount → order completes → dashboard line shows
   discounted price.
3. **Targeted promo on dashboard** — pre-assign `PROMO_DEMO_VIP` to the test
   customer via `bss promo assign` → dashboard surfaces the targeted offer
   card.
4. **Exhausted-code degrade** — exhaust a multi-use code to zero → typed at
   signup → order completes at full price (the v1.1.3 fix).
5. **Step-up auth gate** — sensitive action (payment-method change) requires
   re-OTP; portal challenges and lets through on success.

### Cockpit browser veneer (`localhost:9002`)
6. **Sessions index opens** — `/` lists recent conversations; new-conversation
   CTA navigates to `/cockpit/<id>`.
7. **Tool roundtrip** — operator asks "show me customer alice.demo@bss-cli.local";
   cockpit returns a rendered `customer.get` row.
8. **Propose-then-confirm** — destructive proposal renders a pending banner;
   `/confirm` typed in the input clears it and executes.
9. **Knowledge citation** — "what's our refund policy?" returns an answer
   citing `phases/...` or the doctrine bundle, OR the canonical "I don't have
   a citation for that, try `bss admin knowledge search`" fallback. (Doctrine
   guard 16.)
10. **Slash-command parity smoke** — `/focus CUST-...` and `/clear` both work
    in the browser; mirrors the REPL behaviour.

Out of scope for v1.4.0 (deferred to v1.4.x or later):
- Multi-customer concurrency / parallel session interference
- Cockpit knowledge index rebuild flows
- MNP / port-request UI specs (no portal surface yet)
- CI integration — local-only via `make e2e`

## Design rules

1. **Tests are doctrine-compliant callers.** Setup goes through `bss-clients`
   and the REPL Typer verbs (`bss promo assign`, `bss demo-restore`). No
   direct DB inserts. Asserts may read state via asyncpg for verification, never
   for setup. (Consistent with "write through policy, read freely".)
2. **One demo-prefix convention.** All test artifacts use `*.e2e@bss-cli.local`
   email prefix and `PROMO_E2E_*` ids — surgical cleanup at teardown without
   touching `*.demo@bss-cli.local` (the operator demo data). New prefix because
   demo and e2e have different lifecycles.
3. **No live providers, ever.** `docker-compose.e2e.yml` pins `BSS_PAYMENT_PROVIDER=mock`,
   `BSS_KYC_ALLOW_PREBAKED=true`, `BSS_PORTAL_EMAIL_PROVIDER=logging`,
   `BSS_ESIM_PROVIDER=sim`. Startup template scan that refuses production
   keys with mock providers stays the safety net. (CLAUDE.md anti-pattern
   "no silent fallback when prod creds are missing" still applies — if a real
   Stripe key sits in `.env`, the override pinning `BSS_PAYMENT_PROVIDER=mock`
   would still cause it to refuse to boot; e2e contract is that the user's
   `.env` either has mock creds or the override-only env wins. We document
   the constraint, not enforce it.)
4. **Reports git-ignored.** `docs/e2e-reports/<UTC-timestamp>/` holds the
   pytest HTML report + playwright trace zips + JUnit XML. The dir is in
   `.gitignore`. The README pointer is checked in; the artifacts are not.
5. **Teardown is unconditional.** Makefile target uses `trap` so Ctrl-C and
   pytest crashes still tear down. Override compose stack comes down; main
   compose comes back up; `make seed-demo-reset` purges any e2e-prefix rows
   that survived.
6. **No new write-policy holes.** Step-up gates, ownership guards, escalation
   list — all unchanged. Tests authenticate the same way a real customer
   does (OTP → cookie → session middleware). No bypass shortcuts.

## The `make e2e` contract

```makefile
e2e:  ## Run the v1.4 Playwright suite in mock mode.
	@mkdir -p docs/e2e-reports
	@ts=$$(date -u +%Y%m%dT%H%M%SZ); \
	out="docs/e2e-reports/$$ts"; \
	mkdir -p "$$out"; \
	trap 'echo "--- tearing down e2e stack ---"; \
	      docker compose -f docker-compose.yml -f docker-compose.e2e.yml down --remove-orphans; \
	      $(MAKE) seed-demo-reset >/dev/null 2>&1 || true; \
	      docker compose up -d >/dev/null; \
	      echo "--- e2e report: $$out ---"' EXIT INT TERM; \
	echo "--- bringing up e2e stack ---"; \
	docker compose -f docker-compose.yml -f docker-compose.e2e.yml up -d --wait; \
	$(MAKE) demo-restore; \
	uv run --package bss-e2e pytest packages/bss-e2e/bss_e2e/ \
	    --html="$$out/report.html" --self-contained-html \
	    --junitxml="$$out/junit.xml" \
	    -o playwright_trace_dir="$$out/traces"
```

Two-line UX:
```
$ make e2e
... <test output> ...
--- e2e report: docs/e2e-reports/20260525T133045Z ---
```

## Test data lifecycle

1. `make demo-restore` runs first → clean BSS + loyalty schemas + 3 demo
   customers + the standard demo promos.
2. Each spec's `e2e_customer` fixture creates a fresh `e2e-<uuid>@bss-cli.local`
   identity (so parallel-in-future doesn't collide).
3. `ensure_e2e_promos()` creates `PROMO_E2E_PUBLIC` (5-use code `E2E_PUBLIC10`),
   `PROMO_E2E_TARGETED`, and `PROMO_E2E_EXHAUSTED` (1-use, pre-consumed in
   setup). Idempotent — already-exist returns 'already' per the v1.1.1 verb.
4. At teardown: stack comes down (drops all containers' state if `down -v`
   was used) or `seed-demo-reset` surgical (default). Default is the
   non-destructive variant; full `down -v` requires `make e2e CLEAN=1`.

## Anti-patterns specific to v1.4

- **Don't bypass the OTP flow.** Tests log in via the real magic-link path,
  tailing the `logging` provider mailbox. No direct session-id injection.
- **Don't read from a service's DB schema you don't own.** Cross-domain
  state checks go through `bss-clients`. The only exception is the e2e
  helper's verification reads, which use the public read APIs.
- **Don't hardcode demo IDs.** `PROMO_E2E_*` lives in `helpers/seed.py` as a
  module constant. Same for the test customer email prefix.
- **Don't add a `bss e2e` Typer verb.** The CLI surface stays business-verb,
  per CLAUDE.md ("CLI exposes verbs … not entities"). Makefile is the
  runner.
- **Don't gate tests on real network.** Any spec that depends on Resend /
  Stripe / Didit is by definition wrong for v1.4 — those are channel-layer.

**Non-questions** (settled 2026-05-25): the non-customer → public-code-first
funnel is the intended flow, not a gap. A walk-in types `DEMO_WELCOME10` at
signup, becomes a customer, and only then is eligible for targeted assignment
via `bss promo assign`. The targeted-promo spec in v1.4 mints the customer
first, then assigns — exactly matching how it works in production. No
email-pinned-eligibility design carries to v1.5 from this phase.
